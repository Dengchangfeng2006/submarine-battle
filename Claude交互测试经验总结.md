# 与 Claude Code 交互的测试经验总结

## 一、背景

在潜艇大战游戏的开发过程中，使用 Claude Code 作为代码审查和调试辅助工具，经历了完整的"发现问题→分析假设→验证推翻→定位根因→修复回归"的调试闭环。本文档总结在此过程中获得的交互方法论和测试经验。

---

## 二、核心案例：事件时序 Bug 的三轮排查

### 2.1 问题描述

游戏结束画面按 R 键重开和按 Q 键退出完全无响应。

### 2.2 第一轮：静态代码审查（Claude 主导）

**交互方式**：将完整代码提供给 Claude，要求审查逻辑问题。

**Claude 的输出**：定位到 `showGameOverScreen()` 中 `pygame.event.get()` 的位置问题——在音效播放和渲染之后、`flip()` 之前执行，导致用户在音效期间按下的 R/Q 被当作"残留事件"清空。

**修复**：将 `pygame.event.get()` 移至函数开头。

**结果**：部分改善，但仍不稳定 → **推翻假设，进入第二轮**。

**经验点**：AI 的静态分析能发现明显的逻辑错误（event.get() 位置），但不能保证一次性定位所有问题。单轮分析后必须实际测试验证。

### 2.3 第二轮：机制层面审查（Claude 主导）

**交互方式**：追问"还是按不动"，要求 Claude 从不同角度重新审查。

**Claude 的输出**：发现 `_waitForKey()` 使用 `pygame.time.wait(10)` 轮询模式——Windows 上 Sleep(10) 精度约 10-15ms，事件到达和轮询窗口可能错位。

**修复**：改用 `pygame.event.wait()` 阻塞等待。

**结果**：仍然不稳定 → **再次推翻，进入第三轮**。

**经验点**：AI 容易被引导到"理论上的可能问题"，但实际情况可能更复杂。需要实际测试来验证每个假设。

### 2.4 第三轮：全链路追踪调试（人机协作）

**交互方式**：用户要求"写调试代码"，Claude 在 7 个函数中添加 28 个 `[TRACE]` 追踪点。

**关键发现**：日志输出证明：
- `_waitForKey()` 确实被调用
- KEYDOWN 事件正常到达
- 游戏残留事件被正确过滤（`匹配=False`）
- 用户按键最终被正确检测（`KEYDOWN key=Q, 匹配=True`）

**结论**：第一轮的 `event.get()` 位置修复是核心修复，代码逻辑已正确。

**最终强化**：采用双重检测机制（事件队列 + `pygame.key.get_pressed()` 直接 OS 查询），提供防御深度。

**经验点**：这是最具教育价值的环节。当 AI 的静态分析无法完全定位问题时，**系统性的可观测性设计（TRACE 日志）** 是破局关键。AI 在此步骤中充当了"快速生成追踪代码"的角色，而人类负责解读日志和做最终决策。

## 三、与 AI 交互的测试方法论

### 3.1 AI 能做什么（优势领域）

| 能力 | 表现 | 案例 |
|------|------|------|
| 静态代码审查 | ⭐⭐⭐⭐⭐ | 发现 event.get() 位置错误 |
| 多种实现方案对比 | ⭐⭐⭐⭐⭐ | event.wait() vs time.wait() 轮询 vs get_pressed() |
| 调试代码生成 | ⭐⭐⭐⭐⭐ | 28 个 TRACE 追踪点快速部署 |
| 文档生成 | ⭐⭐⭐⭐⭐ | 实验报告自动生成 |

### 3.2 AI 不能做什么（局限）

| 局限 | 表现 | 应对策略 |
|------|------|----------|
| 无法运行/观察代码 | 只能静态分析，不能看到实际行为 | 必须人工运行 + 查看日志 |
| 倾向于理论化 | 容易陷入"可能有问题"的死循环 | 用 TRACE 日志将猜测转为事实 |
| 需要明确引导 | "没反应"这种描述无法精确诊断 | 用户需要提供具体日志/错误信息 |
| 不熟悉具体运行环境 | 不了解 Windows 11/SDL 2.28.4 的具体行为 | 需要用户在目标环境测试 |

### 3.3 高效交互模式

**模式一：描述问题 + 提供代码**
```
问题：游戏结束画面 R/Q 按键无响应
代码：submarineBattle.py（完整文件）
→ AI 进行静态分析
```

**模式二：追问 + 要求换角度**
```
追问：还是不行，能从其他角度检查吗？
→ AI 从事件管道/控制流/状态机不同维度重新分析
```

**模式三：要求生成调试代码（关键突破）**
```
要求：写调试代码，追踪整个调用链
→ AI 生成 28 个 TRACE 点，将不可见的事件流可视化
```

**模式四：提供日志 + 要求解读**
```
提供：运行时日志输出
→ AI 从日志中提取关键信息，确认修复有效
```

### 3.4 反模式（应避免）

| 反模式 | 问题 | 改进方式 |
|--------|------|----------|
| 多轮猜测式修改 | 没测试就改下一处 | **每轮只改一个变量，测试后再改下一个** |
| 接受所有建议 | AI 的建议可能不适用当前环境 | 批判性审视每个建议的适用性 |
| 不提供上下文 | "没反应"信息量不足 | 提供日志、错误信息、操作步骤 |
| 跳过测试环节 | 改完就认为修好了 | **必须回归测试** |

## 四、测试核心要点

### 4.1 事件处理专项测试

游戏的事件处理系统必须作为**独立测试维度**。验证矩阵：

| | 开始画面 | 游戏中 | 结束画面 |
|---|---|---|---|
| 空格键 | 开始游戏 ✅ | 释放炸弹 ✅ | 忽略 ✅ |
| ← → | 忽略 | 移动军舰 ✅ | 忽略 |
| R 键 | 忽略 | 忽略 | **重新开始** |
| Q 键 | 忽略 | 忽略 | **退出** |
| 关闭窗口 | 正常退出 ✅ | 正常退出 ✅ | 正常退出 ✅ |

### 4.2 时序依赖测试

事件时序相关的代码是**最高风险区域**。需覆盖：

- 渲染过程中按键（event.get() 在渲染前还是后？）
- 音效播放中按键（sound.play() 后立即按？）
- flip() 瞬间按键（显示更新时的事件）
- 快速连续按键（事件堆积）
- 长按不放（KEYUP 延迟到达）

### 4.3 可观测性设计

```
# 调试开关模式
DEBUG_EVENTS = False  # 生产环境关闭

def _waitForKey(self, *targetKeys):
    if DEBUG_EVENTS:
        print(f"[TRACE] 等待按键 {targetKeys}")
    ...
```

建议开发阶段保留追踪代码（通过开关控制），生产环境关闭。

### 4.4 防御深度

关键交互路径应建立多层检测：

```
用户按键
  ├──→ pygame 事件队列 (方法1: event.get())
  │       └──→ 失败？→ 不影响，继续
  └──→ OS 键盘状态 (方法2: key.get_pressed())
          └──→ 独立检测路径，不受事件队列影响
```

### 4.5 回归验证清单

每次修复后必须完成：

- [ ] 开始画面空格键正常
- [ ] 游戏中方向键移动、空格释放炸弹正常
- [ ] 游戏结束画面 R 键重开（测试 3 次）
- [ ] 游戏结束画面 Q 键退出（测试 3 次）
- [ ] 关闭窗口正常退出
- [ ] 连续 3 次完整的"开始→游戏→结束→重开"循环
- [ ] 最高分持久化正常

## 五、关键教训

1. **"按不动"不是简单问题。** 表面是一个按键无响应，背后涉及事件清除时机、轮询机制、OS 键盘状态查询三个层面的交互。永远不要把用户输入问题简单化。

2. **先观测，后修改。** 最有价值的一步不是任何代码修复，而是添加 28 个 TRACE 追踪点将事件流可视化。在没有观测手段的情况下修改代码 = 盲人摸象。

3. **AI 是加速器，不是替代品。** Claude 能快速生成调试代码、从多角度分析、自动生成文档，但最终判断（哪个修复真正有效、是否需要双重检测）必须由人类做出。

4. **每个修复都要回归测试。** 即使一个修复"看起来正确"，也必须实际运行验证。event.get() 位置修复在理论上是完美的，但只有配合双重检测才达到 100% 可靠。

5. **防御深度是工程智慧。** 单一检测机制存在单点故障风险。pygame.key.get_pressed() 直接从 OS 查询键盘是一条完全独立的路径——即使 SDL 事件系统在特定平台/版本下行为异常，备用路径仍能保证功能正常。
